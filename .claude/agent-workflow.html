product-ux 「何を作るか」を決める人

「仕様を考えながらコードも書く」は、料理のレシピを考えながら同時に調理するようなもの。両方中途半端になるため分ける。

この人が決めること

  • 誰のための画面か（ペルソナ）
  • 何を一番見せるか（優先順位）
  • どう操作するか（画面フロー）
  • どんな言葉を使うか（文言）

出力はコードではなく HTMLモックと指示書。実装はしない。

snow-data 「この数字は何を意味するか」を定義する人

積雪50cmが「山頂だけ」なのか「全体の平均」なのかで、ユーザーの判断が180度変わる。データの意味を間違えたまま実装すると致命的なミスになる。

この人が決めること

  • 積雪/降雪の定義（どこで測った何の数値か）
  • ゲレンデとデータの対応表
  • 「この数値はこう表示してはいけない」という境界線

コード編集不可。data-dictionary.md / resort-mapping.md に出力する。

architecture 「どこを触るか」の地図を描く人

「積雪表示を直して」と言うだけでは、100行あるコードのどこを触ればいいかわからない。間違った場所を触って別のものが壊れる。

この人が決めること

  • 触るべき関数・ブロックの場所
  • Leafletのどのレイヤーが何をしているか
  • Aを変えるとBが壊れる、という依存関係

コードは書かない。地図だけ渡す。 データの正しさの定義はsnow-dataに寄せる。

qa-accessibility 「ちゃんと動くか・誰でも使えるか」を確認する人

implementerは「動くコードを書く」のが仕事。自分で書いたものの欠陥は自分では見つけにくい。別の目で見る人が必要。

この人が確認すること

  • スマホ・PCで見え方がおかしくないか
  • 手袋でもタップできるか（44px以上）
  • 色のコントラストは十分か
  • バグの再現手順

コード編集不可。[QA REPORT] 形式で出力 → 差し戻しまたはdocs-knowledgeに記録。

docs-knowledge 「何を決めたか」を記録する人

会話を閉じると記憶が消える。「あのとき何を決めたっけ」が積み重なると、同じ議論を何度もすることになる。

この人がやること

  • 各エージェントのHandoffを受け取って整形
  • 決定事項を1行ログとして保存
  • 次の会話の冒頭に渡す引き継ぎメモを作る

コード編集不可。decision-log.md / changelog-draft.md に出力する。

implementer 「手を動かす」唯一の人

全員がコードを触ると、誰かが誰かの変更を上書きする。コード編集の窓口を1人に絞ることで事故を防ぐ。

この人のルール

  • 「実装して」と言われたタスクだけやる
  • 仕様は他のエージェントが決めたものを受け取る
  • 自分では仕様を決めない
