## 1. 問題定義（何が問題か）

| ID | 観察 | なぜ問題か（B向け） |
|----|------|---------------------|
| P1 | **情報階層がフラット** | 降雪・雨・風が同じテキスト粒度。「明日行けるか」の答えが先頭にない。 |
| P2 | **積雪が「説明文」扱い** | 0.0cm でも視覚的に目立たず、スキャンして数字を探す必要がある。 |
| P3 | **日付チップ下が「—」だらけ** | 横スクロールの意味（どの日が良いか）が一目で伝わらない。 |
| P4 | **タイムラインと注釈の重複** | 「D-1予測」系ラベルがセル付近に重なり、誤読・ストレスになる（レイアウトバグ相当）。 |
| P5 | **Primary 操作が不明** | 上部の「次→」が詳細文脈では目的不明。閉じる・日付変更・別スポットの優先が読めない。 |
| P6 | **A/B 未分離** | 比較・根拠が欲しいAと、即断したいBが同一UIで、Bにとってノイズが多い。 |

## 2. 新しい画面フロー（提案）

```text
[マップ]
  └ ピンタップ
       └ ボトムシート起動（デフォルト: 直前モード=B）
            ├ 1画面目（Bコア）
            │    ├ ツールバー: [直前モード ON/OFF] … [閉じる]
            │    ├ スポット名・メタ（小）
            │    ├ ヒーロー: 明日のゴー判定（文言+色）
            │    ├ 積雪: 前日合計 / 当日合計（大数字2カード）
            │    ├ 雨リスク1行 + 風1行
            │    ├ 日付チップ（各チップにその日の降雪見込みを必ず表示）
            │    ├ タイムライン（ラベル行とアイコン行を分離・注釈は上段のみ）
            │    └ フッター: [シートを閉じる] Primary / [別のゲレンデ] Secondary
            │
            └ 直前モード OFF（A）
                 └ ヒーロー折りたたみ or 比較表へ遷移（既存リスト/地図連携は実装側で既存フローに接続）
```

**操作優先順位（確定案）**

1. **閉じる**（誤開き・確認完了）  
2. **対象日**（デフォルトは「明日」にフォーカス）  
3. **直前モード ⇄ 計画モード**  
4. **別スポット**（探索は二次）

**代替案**

- **代替A**: マップ上に「明日サマリー」ピンバッジ（タップ前から一行判断）— 開発コストと誤読リスクのトレードオフあり。  
- **代替B**: B専用の「今日の一行カード」ホームから行き先直リンク（マップは二次）。

## 3. 仕様・受け入れ条件（モック準拠）

- **3秒判断**: シートを開いて **1秒以内**にヒーロー帯で Go/Caution/No-Go（または同等の一言判定）が読めること。  
- **積雪**: 「前日合計」「当日合計」は **28px級以上**（モック参照）で、ラベルより明確に大きいこと。  
- **日付チップ**: 降雪見込みが無い場合でも **「0 cm」またはデータ欠の明示**（裸の「—」のみは避ける）。  
- **タイムライン**: 可変長ラベルは **専用行**に置き、セル上に重ねない。  
- **CTA**: 「次→」に相当する文言は **使わない**。別スポットは Secondary 以下。  
- **トークン**: 本番は `ski-powder-hunter.html` の CSS 変数に合わせること。
