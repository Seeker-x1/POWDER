# 実装指示書：汎用表示の整理 ＋ 前日実績（気象庁）の表示

**対象:** `ski-powder-hunter.html` / `ski-powder-hunter-en.html`（日英とも同趣旨で揃える）  
**関連ポリシー:** `docs/MAP_UI_OVERLAY_POLICY.md` / `docs/MOBILE_MAP_UI_IMPLEMENTATION_BRIEF.md`（マップ枠組みは別紙）

---

## 0. 目的（プロダクト）

1. **PC まわりの「汎用」に見える説明・重複 UI を削り**、画面をスキー場利用の文脈に寄せる。  
2. **「今日」選択時**に、**気象庁の実績**に基づく **前日の降雪情報**を出す（Open-Meteo は **前日実績のソースに使わない**）。  
3. **降水量＋気温条件による降雪相当量の解釈は必須**（全地点・全表示で同一ルール）。

## 1. 汎用表示の削除・整理

### 1.1 コード上の注意

リポジトリ検索では **HTML 内に「汎用」という字面は出てこない**。実装前に **ディレクターとスクリーンショットで「消す対象」を確定**すること。  
以下は **よくある候補**（`ski-powder-hunter.html` 基準の行番号は目安。変更されていたら grep で再特定）。

| 候補 | 場所（目安） | 指示 |
|------|----------------|------|
| **パウダースコアの短い一般注釈**（「3〜4日後がおすすめ。予報は参考値。」） | `#info-banner` 先頭 `<strong>` 行付近（現状 ~762 行） | **該当文を削除**するか、**折りたたみ内のみ**に移すかはディレクター判断。本指示のデフォルトは **先頭の強い一行を削除**。 |
| **折りたたみ詳細**（`info-banner-detail`） | 同ブロック内 | 先頭一行削除に合わせ、**重複・冗長**なら整理。完全削除は任意。 |
| **補助ヒント括弧** | `#sort-depth-hint`（~777 行） | 「汎用説明が長い」場合は **非表示または文言短縮**。 |
| **トップのフォールバック表記** | `renderTop10` 付近「・フォールバック」、`フォールバックデータ` | ユーザー向けに **汎用ラベルが目立つ**なら、**文言の一般化を避ける**（例: 技術用語を出さない・1語に抑える）。 |

### 1.2 重複 UI（前日／当日／翌日の大きい二枚）

サイドバーカード内 **`pw-three-day-strip`**（`前日` / `当日` / `翌日` の並び）が、**日別棒グラフやサマリーと数値重複**している場合は **削除または折りたたみ**する（ディレクター判断済みなら **削除でよい**）。

### 1.3 英語版

`ski-powder-hunter-en.html` の **対応する `info-banner` 文言・strip・ヒント**を同様に処理する。

### 1.4 受け入れ（汎用）

- [ ] PC 幅で、**「3〜4日後がおすすめ…」が不要なら表示されていない**  
- [ ] **当日／翌日の重複カード**が方針どおり消えている（または役割が明確に分かれている）  
- [ ] 日英で **同じ情報設計**

## 2. 前日実績（気象庁）— データ方針（必須）

### 2.1 ソース

- **前日合計（JST カレンダー前日 0:00〜24:00）**  
- **前日 12–18 時**（JST 前日 12:00〜18:00 の合計）  
- **前日 18–24 時**（JST 前日 18:00〜24:00 の合計）  

上記はすべて **気象庁の観測系データ**から算出する。**Open-Meteo の hourly で昨日を埋めない。**

### 2.2 降雪量の定義（必須）

- **降水量（時別）＋気温（時別）**から、**実装で固定した閾値・式**で「降雪相当 cm」に変換する。  
- ルールは **コード＋短いユーザー向け注記**（例: 「気象庁観測・降水と気温から推計」）で共有可能にする。  
- 欠測時は **セルを空／「—」**、必要ならツールチップで理由。

### 2.3 地点

- 各ゲレンデに紐づく **代表観測点**（既存 `jma` 連携と同系統でよい）を使う。  
- 地点が取れないゲレンデは **前日実績セクションを非表示**または「—」。

### 2.4 配信・キャッシュ

- ブラウザが気象庁公式ページを都度スクレイプする実装は **避ける**（不安定・規約リスク）。  
- **サーバ取得＋ JSON 化**、または **ビルド／定期ジョブで `data/` に書き出し**など、**再現可能なパイプライン**にする。  
- 「昨日」の時系列は **カレンダー固定**なので、**1 日 1 回以上**前日分が揃えばよい（頻度は運用で決定）。

## 3. UI 仕様（デザイン合意）

### 3.1 日別タブ「今日」選択時

- **前日合計**を、**今日タブの左側（横並び）**に表示する（モック・ディレクター指示どおり）。  
- ラベル例: **`前日合計（気象庁） … cm`**（文言は日英化）。

### 3.2 時間帯別バー（2 枚目相当の帯）

- 既存の **「0–5 時」…「翌日…」** の **左端**に、次の **2 本**を **この順**で追加する。  
  1. **`前日 12–18時`**  
  2. **`前日 18–24時`**  
- 続けて **当日 0–5 時** … と **左→右が時系列**になること。  
- ラベルは **JST カレンダー前日に固定**（閲覧時刻で意味が変わらない）。

### 3.3 既存 Open-Meteo 帯との関係

- **今日以降の予報帯**は従来どおり Open-Meteo でもよい。  
- **前日の 2 本＋前日合計**だけ **気象庁実績＋降水×気温ルール**。  
- UI 上 **データソースを混在させる場合**は、小さく **（気象庁）／（予報）** 等で区別できるとよい。

## 4. 実装上の既知論点（参考）

- `getDayIndex` と `dayIdx>=1` 条件により、**「今日」の「前夜〜朝」が 0 になりやすい**経路があった。**本指示の「前日 12–18 / 18–24」は JMA 実績で別経路供給**とし、既存の予報ベースの「前夜〜朝」と **二重表示にならないよう**整理する（どちらを残すかは **実績優先**でよい）。  
- モバイル用 `selectResort` 内の時間帯 `segments` も、**同じラベル・同じデータソース方針**に揃えるか、**PC/モバイルで一関数に集約**を推奨。
