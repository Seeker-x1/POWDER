# 実装指示書：JMA 降雪メッシュ「一コマおきに消える」補完

## 0. 現象

- 地図の **ズームを1段ずつ変えると、降雪メッシュが表示されたり消えたりする**（コマ送りで不安定）。
- いまの実装は `L.tileLayer` の **`maxNativeZoom` / `minNativeZoom`** だけで、**「取れない z を別の z のタイルで補う」専用ロジックはない**。

**参照コード（目安・行番号は変わりうる）**

- `ski-powder-hunter.html`：`ensureJmaSnowLayer` 内 `L.tileLayer(urlPattern,{ … maxNativeZoom:8, minNativeZoom:6 … })`
- 同：`getJmaSnowEffectiveTileZoom` / `getJmaSnowAlternateProbeZ`、`probeJmaSnowTileExists` / `isJmaSnowTimeRenderable`（**表示と同じ z 規則に揃える必要あり**）※実装済み: `JmaSnowTileLayer._clampZoom` で偶数 z に揃え、プローブも同一関数を使用
- `ski-powder-hunter-en.html` も **同一変更**

---

## 1. 目的（完了の定義）

1. **ユーザー操作ズーム（例: 3〜10）の各整数レベル**で、可能な限り **メッシュが「真っ白で欠ける」状態にならない**。  
2. サーバに **実在しない z** を叩き続けない（404 の繰り返しを減らす）。  
3. **時刻プローブ・自動更新**が、**補完後のタイル z 規則**と矛盾しない（切り替え直後にレイヤだけ消える、を防ぐ）。

---

## 2. 実装前の調査（必須）

1. **開発者ツールの Network** で、降雪 ON・固定 `timeStr` のとき、**同じ地点のタイル URL を z=5,6,7,8,9… と手で変えて** 200 / 404 を表にする（または既存 `probe` ログを流用）。  
2. **「奇数だけダメ」「偶数だけダメ」「8以上すべてダメ」**などパターンを **事実として記録**する。  
3. その結果に基づき、§3 の **どの案を採るか**を決める（推測で実装しない）。

---

## 3. 実装案（優先順）

### 案 A：`zoomOffset`（最優先で試す・変更が最小）

Leaflet の `L.TileLayer` は **`zoomOffset`** で「地図ズームとタイル URL の z」のずれを一括指定できる。

- 例：**地図 z=7 で取れないが z=6 で取れる**なら、`zoomOffset: -1` などで **常に 1 段低いタイルを引き、拡大表示**させる、といった調整が可能。  
- **効果が足りなければ案 B へ**。

**注意:** `minNativeZoom` / `maxNativeZoom` と **同時に**意味が変わるため、調整は **セットで**試す。

---

### 案 B：タイル z の「補完関数」（奇数→偶数など）

JMA 側が **特定の z だけ欠ける**場合、**リクエスト z を常に「取れる側」に丸める**。

例（概念・数値は調査結果で決定）:

- `z_req = (z % 2 === 1) ? z - 1 : z` のように **奇数ズームのときだけ 1 段下げる**  
- または `z_req = Math.min(z, Z_MAX_REAL)` で **上限クリップ**

**実装方法（推奨）:**

- `L.TileLayer` を **継承**し、`getTileUrl(coords)`（またはプロジェクトの Leaflet バージョンに合わせた同等フック）で  
  **`coords` の (x,y,z) を `z_req` に合わせて親タイル座標へ変換**してから URL を組み立てる。  
  - z を 1 下げるごとに `x = Math.floor(x/2)`, `y = Math.floor(y/2)` を繰り返す（標準的なタイルピラミッド）。

**禁止:** URL の `{z}` だけ差し替えて **`x,y` をそのまま**にすること（座標系が壊れてランダムに欠ける）。

---

### 案 C：`GridLayer` 全面カスタム

案 A/B で足りない場合のみ。コストが高いので **最後の手段**とする。

---

## 4. プローブとの整合（必須）

`fetchJmaSnowLatestTime` / `isJmaSnowTimeRenderable` / `probeJmaSnowTileExists` は **「実際にレイヤが表示に使う z」**でヒット判定すること。

- レイヤが **常に z-1 のタイルを使う**なら、プローブも **同じ z-1**（または同じ `z_req` 関数）で試す。  
- **`JMA_PROBE_Z` 固定のまま**レイヤだけ `zoomOffset` するのは **不整合になりやすい**（時刻は取れたのに画面だけ真っ白、等）。

---

## 5. オプション・UX

- **高ズームは意図的にぼかす**（拡大タイル）なら、**ユーザー向けに一言も不要**なことも多い。  
- もし「なぜ荒いか」を出すなら、**既存の注釈方針**（地図上に常時パネルを増やさない）に従う。

---

## 6. 受け入れ条件

- [ ] 降雪レイヤ ON で、**同一地点・同一時刻**のまま **ズームを 3→4→…→10 と一歩ずつ変え**、**連続で真っ白にならない**（許容: 極端に縮小したときのみ非表示など、**事前に文書化した範囲**）  
- [ ] **奇数 / 偶数**どちらのステップでも、調査で決めた方針どおり **補完が効いている**  
- [ ] **時刻の自動更新・手動切替**後もメッシュが安定  
- [ ] 日英 HTML **両方**で動作一致
