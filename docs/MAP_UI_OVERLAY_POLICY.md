# マップ周りオーバーレイ・凡例のポリシー（汎用指示）

**対象:** `ski-powder-hunter.html` / `ski-powder-hunter-en.html` および同種のマップUI。  
**読者:** 実装者・レビュー・AIエージェント（指示の取り違え防止用）。

---

## 1. 汎用指示（デフォルト）

1. **地図キャンバス上に、常時表示の「凡例」「色分け説明」「ランキング意味のパネル」を複数置かない。**  
   ユーザやプロダクトが **明示的に「常時表示で欲しい」** とした場合のみ例外。それ以外は **非表示**、または **ユーザーが意図して開く1つのUI** にまとめる。

2. **気象庁タイル等の「降雪スケール凡例」は、地図に独立したフロートボックスとして重ねない。**  
   必要なら **レイヤーON時のみ**、かつ **レイヤー操作と同じブロック**（例: トグル直下の折りたたみ、レイヤー専用シート、ヘルプ）の **設計済みの1か所** に限定する。  
   （旧 `#jma-legend` はこの方針で削除済み。別ID・別位置での **实质的な復活** は禁止。）

3. **ピン色・風速帯など既存の `.map-overlay` 系も「地図の主役」を損なうなら、同様にオンデマンド化またはサイドバー統合を検討する。**  
   デスクトップのみ表示・モバイルでは非表示など、**ブレークポイントで役割を分けてよい**。

---

## 2. 理由（なぜ消す／重ねないか）

| 理由 | 説明 |
|------|------|
| **地図が主役** | 本アプリの価値は「位置とピン」。説明パネルが地図を覆うと探索体験が損なわれる。 |
| **複数フロートの破綻** | 凡例A（ピン）＋凡例B（JMA）＋トグルが **それぞれ独立** すると、**パッチワークUI** になり、どれも小さく・どれも邪魔、という最悪パターンになりやすい。 |
| **モバイルとの競合** | 検索・戻る・JMA帯・ボトムシートと **z-index・タップ領域** が衝突し、**閉じられない／パンできない** 事故の温床になる（別ブリーフの状態モデルと整合させる）。 |
| **「ずらすだけ」はレイアウトではない** | 重なりを **座標（left/bottom）だけ変更** で回避すると、別の重要領域（ズーム、南海、シート、ノッチ）を侵食し、**次の要素とまた衝突**する。デザイン上の **優先順位・情報の階層** が決まっていないままの移動は **不採用**。 |

---

## 3. 禁止する実装パターン

- **「かぶったから動かした」だけの対応**（どこに置くべきかの設計・合意なし）。  
- **凡例を増やして読みにくさを「説明文」で解決しようとすること**（認知負荷が増えるだけ）。  
- ドキュメントで **削除／常時オーバーレイ禁止** と書いた要素を、**別名DOM・別クラスで同じことをさせること**（实质的復活）。  
- オーバーレイを足すとき **状態モデル（地図のみ／詳細シート／検索フォーカス等）を無視** すること。

---

## 4. 許容する代替（必要な説明の出し方）

- **ヘルプ・About・「？」から開くモーダル／ボトムシート** に集約。  
- **レイヤーON時だけ**、トグル付近の **折りたたみ1本** にスケールを書く（地図上に第2の箱を増やさない）。  
- デスクトップ限定の **サイドバー内** 説明（地図と競合しない領域）。  
- ツールチップは **控えめに**（モバイルは長押し・シートとの兼ね合いに注意）。

---

## 5. レビュー時のチェック質問

- 地図上に **説明用の箱が2つ以上** 同時に出ていないか。  
- 新しいフロートは **どの状態（詳細シート開閉等）で隠す／下げる** か決まっているか。  
- 位置調整が **「衝突回避の座標パッチ」** になっていないか。
