# Japowserch（JAPOWSERCH）ロードマップ・SEO・不足機能プラン

## 1. 現状整理

### 競合との差別化（強み）
| 項目 | powdersearch.jp | snow-forecast.com | 自サイト（現状） |
|------|-----------------|-------------------|------------------|
| データソース | 独自・GSM等 | 山岳気象モデル | **Open-Meteo + 気象庁**（積雪・3h観測） |
| 営業終了後の降雪 | 不明 | 前日まで降雪を表示 | **気象庁で「昨日の夜〜今朝」を補強可能** |
| 日本ゲレンデ数 | 多数 | 3300リゾート（世界） | **約450ゲレンデに特化** |
| モバイル | テーブル中心 | 汎用 | トップ＋マップ切替あり |
| エリア絞り | あり | 国・地域 | マップ画面のみ（**トップにはなし**） |
| マップで複数比較 | - | - | **モバイルで1ピンのみ・一覧で探しづらい** |

### サイトの強み（訴求すべきポイント）
- **「今日行く」判断**: 気象庁の観測（積雪・直近3h降雪・日最深）で「昨日の営業終了後にどれだけ降ったか」が分かる設計にできる。
- **雨・風・気温**: 予報で雨確率・山頂風・気温を出しており、リフト止まり・雪→雨を避ける用途と合致。
- **無料・日本語・国内特化**: 会員登録なしで全機能使える。

## 2. 自サイトに不足しているもの（提案）

### 2.1 SEO・発見されやすさ
- **meta description / OGP**: 未設定。検索結果・SNSシェアで説明文が出ない。
- **タイトル**: 「Japowserch」のみ。キーワード「パウダー ランキング」「スキー場 雪 予報」「ゲレンデ 積雪」等が含まれていない。
- **構造化データ**: スキーリゾートの `LocalBusiness` / `SportsActivityLocation` や、ランキングページの `ItemList` が無い。
- **コンテンツページ**: トップと1枚のマップのみ。競合は「地域別」「リゾート別」などページが多く、ロングテールで流入している。
- **canonical / hreflang**: 日本語・英語版の関係が検索エンジンに伝わっていない可能性。

### 2.2 データ・情報
- **「昨日の営業終了後〜今朝」の降雪**: 気象庁の3h観測や日最深はあるが、「昨夜〜今朝の合計降雪」を**今日行く判断用**に分かりやすく出していない。  
  → 気象庁の観測値から「前日18時〜当日6時」などの区間降雪を算出・表示するブロックを追加すると差別化になる。
- **運行状況の一元化**: 各ゲレンデの公式が発信している運行・積雪を「ここを見れば分かる」にしていくには、**運営者投稿用の窓口**（フォーム・連絡先・将来的には簡易CMS）が必要。

### 2.3 UX・導線
- **トップのエリア絞り**: 現在は全国TOP10のみ。東京から行ける「関東・長野・新潟」などに絞ったランキングができない。  
  → コンパクトなエリア選択（ドロワー／タブ／1行ピル）をトップに追加。
- **マップで「次/前」ゲレンデ**: モバイルでマップのみ表示のとき、1ピンだけ開いた状態から**ごちゃつかずに**別ゲレンデへ移る手段（次へ/前へボタン、またはスワイプで切り替え）がない。
- **利用シーン別の入口**:  
  - 数日後に行く（週末計画）→ 日付＋エリアでランキング。  
  - 当日・前日に決める → 「昨夜〜今朝の降雪」＋積雪＋風・気温。  
  - 泊の参考 → エリア別ランキング＋リンク（宿検索は外部でよい）。  
  これらが「ワンタップ/少ない操作」で選べるようにする。

### 2.4 信頼・運用
- **データの出所と更新時刻**: 「気象庁」「Open-Meteo」の明記はあるが、「いつ時点の観測か」「予報の信頼度（1〜2日/3〜5日）」をトップや凡例でもう一歩はっきり書く。
- **ゲレンデ運営者向け案内**: 「運行状況・積雪をここに書いてもらう」ためには、「運営者向け」ページ（掲載条件・連絡先・どんな情報を載せられるか）が必要。

## 3. ここまで行くまでのステップ（プランニング）

### Phase 0: 土台（SEO・メタ・コンテンツ）
1. **メタ・OGPの追加**  
   - `<title>`: 例）`パウダーランキング｜スキー場の雪・積雪予報【Japowserch】`  
   - `<meta name="description">`: 80〜120字で「日付を選んでその日のパウダースコアTOP10」「気象庁＋Open-Meteo」「雨・風・積雪で行き先を比較」を記載。  
   - OGP: `og:title`, `og:description`, `og:image`, `og:url`, `og:type`。
2. **構造化データ**  
   - トップ: `ItemList`（ランキング）。  
   - 各リゾート（マップポップアップ or 将来的な詳細ページ）: `Place` / `SportsActivityLocation`、名前・住所・緯度経度・URL。
3. **日本語/英語の関係**  
   - 同一URLで `hreflang` を使うか、`ski-powder-hunter.html`（ja）と `ski-powder-hunter-en.html`（en）で `hreflang` を相互指定。

### Phase 1: トップの「エリア絞り」（ごちゃつかせない）
4. **トップにエリア選択を追加**  
   - **案A**: 日付の下に「全国」「北海道」「東北」「新潟」「長野」…の**1行スクロールピル**（マップの region-pills と同一値）。選択で `renderTop10()` の対象を `RESORTS` のうち `region` でフィルタし、そのエリアのTOP10を表示。  
   - **案B**: 「エリアで絞る」ボタン1つで、コンパクトなドロワーまたはモーダルを開き、地域を選ぶ。  
   - デフォルトは「全国」。見た目はトップの余白をなるべく使わず、1行＋小さいフォントで収める。
5. **TOP10の見出しを動的に**  
   - 例）「3月18日のパウダースコア TOP10（全国）」→ エリア選択時は「（関東）」などに変更。

### Phase 2: マップで「数日後に行く場所を探す」のに向いた操作（モバイル）
6. **マップ上の「次/前」ゲレンデボタン**  
   - ポップアップを開いているとき、**ポップアップ内**または**マップ右上/下のフロート**に「← 前のゲレンデ」「次のゲレンデ →」を配置。  
   - 対象は「現在のソート順（パウダースコア順など）のフィルタ結果」の前後。これでマップを動かさずに一覧性を保ったまま複数ゲレンデを比較できる。
7. **モバイルで「リストをチラ見」**  
   - マップ画面で、下端に「リスト」タブまたはスワイプで少しだけリストを引き上げる**ハーフシート**を用意し、スクロールでゲレンデを選ぶとマップのピンとポップアップが連動。ごちゃつかない程度にリストを出せる。

### Phase 3: 「今日行く」判断の強化（気象庁の活かし方）
8. **「昨夜〜今朝の降雪」表示**  
   - 気象庁の観測（3h降雪やアメダス等）から、**前日営業終了想定時刻〜当日朝**の区間降雪を算出するロジックを用意（バックエンド/スクレイピング/気象庁APIの可否を調査）。  
   - トップのカード・マップのポップアップ・サイドバーのカードに「昨夜〜今朝 +○cm」のようなバッジを追加。
9. **運用ルールの明文化**  
   - 「積雪・降雪は気象庁○○観測点を参照」「更新は毎日○時頃」を「詳しく」やフッターに記載。

### Phase 4: 利用シーン・泊の参考
10. **トップの短文案内**  
    - 「数日後の休みにどこ行く？」「当日朝、どこが良さそう？」「風が弱くてリフト止まりにくいところ」を1行ずつ＋該当する操作（日付選択・エリア・ソート）へ誘導。
11. **エリア別ランキングと宿**  
    - エリアTOP10の近くに「このエリアの宿を検索」などで楽天・じゃらん等へのリンク（アフィリエイト可）。泊の参考になることを明示。

### Phase 5: ゲレンデ運営者が「ここに書けば見てもらえる」状態へ
12. **運営者向けページ**  
    - 「ゲレンデ運営者・施設の方へ」ページを作成。掲載しているゲレンデ一覧、運行状況・積雪・臨時休業などを「ここに報告してもらうと掲載する」旨と連絡方法（メール/フォーム）を記載。
13. **報告フォーム**  
    - ゲレンデID/名前、項目（営業有無・リフト本数・積雪・コメント）、日時を入力して送信。管理側で確認して `resort-operating.json` や `resort-snow.json` を更新（手動または簡易バックエンド）。
14. **「運営者報告」バッジ**  
    - ゲレンデカードやポップアップに「○月○日 運営者報告」のような表記を入れ、ユーザーに「公式に近い情報」と分からせる。
15. **徐々に公営・古いゲレンデへ**  
    - まずは発信しているゲレンデから掲載情報を集め、同じ地域の公営・小規模ゲレンデに「こちらでも報告できます」と案内を広げる。

## 4. 実装の優先度（目安）

| 優先度 | 内容 | 目的 |
|--------|------|------|
| 高 | メタ・OGP・title/description | 検索結果でのクリック率・SNS |
| 高 | トップのエリア絞り（1行ピル or 1ボタン） | 東京など「行ける範囲」でランキングを見たいニーズ |
| 高 | マップの「次/前」ボタン（＋ソート順連動） | モバイルで複数ゲレンデを比較しやすくする |
| 中 | 構造化データ | リッチリザルト・地域検索 |
| 中 | 「昨夜〜今朝の降雪」表示 | 当日朝の判断の差別化 |
| 中 | トップの利用シーン案内（1行×3） | 初回訪問者の迷いを減らす |
| 低 | リストハーフシート（モバイル） | マップとリストの両立 |
| 低 | 運営者向けページ＋フォーム | UGC・運営者投稿の受け皿 |
| 低 | 地域別静的ページ（将来） | ロングテールSEO |

## 5. 競合上位化のために意識すること

- **キーワード**: 「パウダー ランキング」「スキー場 雪 予報」「ゲレンデ 積雪 今日」「スキー どこ 雪」などを title / description / 見出しに自然に含める。
- **更新頻度**: 毎日更新される「今日のランキング」であることを、更新時刻表示と description で伝える。
- **被リンク**: スキー系メディア・ブログ・SNSで「今日のパウダーはこのサイトで見る」と紹介してもらえるよう、URL・OGPを整えておく。
- **体験の一言**: 「日付とエリアを選ぶだけ」「気象庁の観測で昨夜の降雪が分かる」など、一言で説明できる売りをトップとOGPに書く。

## 6. まとめ

- **不足している主なもの**: SEOメタ・OGP、トップのエリア絞り、マップで複数ゲレンデを探す操作（次/前ボタン）、「昨夜〜今朝の降雪」の明示、運営者投稿の受け皿。
- **ステップ**: Phase0でSEO土台 → Phase1でトップのエリア絞り → Phase2でマップの次/前 → Phase3で気象庁の「営業終了後」降雪 → Phase4で利用シーン・泊の参考 → Phase5で運営者投稿・公営ゲレンデまで広げる。
- まずは **Phase 0 と Phase 1** を実装すると、検索での露出と「東京からどこ行くか」の使い勝手が一気に良くなる。
