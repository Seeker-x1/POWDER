# ゲレンデリストの追加と公式リンク確認

## 網羅できていないゲレンデを追加する

### データの場所
ゲレンデ一覧は **ski-powder-hunter.html** 内の **`const RESORTS = [ ... ];`** に格納されています。

### 1件分の形式（必須項目）
```json
{
  "id": 451,
  "name": "ゲレンデ名",
  "pref": "都道府県名",
  "region": "hokkaido|tohoku|niigata|nagano|chubu|kanto|kansai|chugoku|shikoku",
  "lat": 緯度（数値）,
  "lng": 経度（数値）,
  "url": "公式サイトのURL（なければ空文字 \"\"）",
  "elevation": { "base": 麓標高m, "top": 山頂標高m },
  "tags": ["powder"] または ["groomed"] など
}
```

### region の値
| 値 | 地域 |
|----|------|
| hokkaido | 北海道 |
| tohoku | 東北 |
| niigata | 新潟 |
| nagano | 長野 |
| chubu | 中部（富山・岐阜・石川等） |
| kanto | 関東 |
| kansai | 関西 |
| chugoku | 中国 |
| shikoku | 四国 |

### 追加の手順
1. **id** は既存の最大 id より大きい連番（例: 451, 452 …）にする。
2. **name** は公式表記に近い名前にする。
3. **lat / lng** はゲレンデ中心またはリフト上端付近の緯度経度（Google Maps 等で確認）。
4. **url** は公式サイトのトップまたはスキー場案内ページ。不明な場合は `""`。
5. **elevation** が不明な場合は `{ "base": 0, "top": 500 }` など仮の値でも可。
6. **tags** は `["powder"]`（パウダー向け）, `["groomed"]`, `["kids"]`, `["backcountry"]` など。複数可。
7. 既存の配列の **最後の `}` の直後にカンマを付け**、上記形式のオブジェクトを 1 件追加する。

### 注目ランキングに含めたい場合
**FEATURED_IDS** に、追加したゲレンデの **id** を追加すると、地図の「注目のみ」表示やトップの候補に含まれます。

## 営業期間（openDate / closeDate）の確認

### 現在のやり方
営業日は**自動取得していません**。次の2つだけを見ています。

1. **RESORTS の各オブジェクト**  
   `openDate`（営業開始日）と `closeDate`（営業終了日）を **"MM-DD"** または **"MM/DD"** で持てます。  
   例: `"openDate":"12-01","closeDate":"04-30"`
2. **data/resort-operating.json**（あれば）  
   `{ "リゾートID": { "open": "12-01", "close": "04-30" }, ... }` の形。  
   ここに書いた値が、RESORTS の openDate/closeDate より**優先**されます。

どちらも**手動で更新**する前提です。公式サイトやニュースで「〇月〇日営業終了」を確認したら、RESORTS または resort-operating.json の closeDate をその日付に合わせて更新します。

### 名前の表記について
公式サイトでは「南**ケ**丘スキー場」、アプリでは「置戸町南**ヶ**丘スキー場」のように、町名付き・表記ゆれ（ヶ/ケ）が出ることがあります。運用では「同じゲレンデ」と判断して問題ありません。必要なら RESORTS の `name` を公式表記に寄せて更新してください。

### 一つの方法で揃えるのは難しい場合の進め方
公式サイトの作りはゲレンデごとにバラバラ（HTML、PDF、画像、ニュース告知など）なので、**全件を同じ仕組みで自動取得するのは現実的ではありません**。次のような**組み合わせ**がおすすめです。

1. **運用でカバー（現実的）**
   - シーズン前に「営業終了が早いゲレンデ」「営業終了した」のリストをチェックし、**resort-operating.json だけ**を更新する運用にする。
   - RESORTS の openDate/closeDate は「だいたいの期間」の初期値にして、**細かい変更は resort-operating.json に集約**すると差分管理しやすい。

2. **公式URLを活用**
   - 各ゲレンデの `url` を公式の営業案内ページにしておく。
   - 定期的に「営業終了」「〇月〇日まで」などのキーワード＋日付をスクレイプする**補助スクリプト**を用意し、**候補だけ抽出 → 手動で確定**する流れにすると、更新漏れを減らせる。

3. **まとめサイトを補助に**
   - gelanding などまとめサイトから期間を取得する方法もあるが、**一次情報は公式**。公式と矛盾する場合は公式を優先し、まとめは「候補の洗い出し」程度に使う。

4. **resort-operating.json を主軸にする**
   - 上記のように「更新は JSON に集約」にすると、スクリプトで JSON を生成・更新しやすくなる。  
   - 例: 公式URL一覧を元にスクレイプ結果を JSON にマージするスクリプトを用意し、実行後に目視で確認してからコミットする。

## 公式サイトリンクの確認

### 方法1: ブラウザで手動確認
- マップまたはリストでゲレンデをクリック → ポップアップの「公式 ↗」リンクを開く。
- 404 や別サイトに飛ぶ場合は、正しい URL を調べて RESORTS の `url` を修正する。

### 方法2: リンク確認スクリプト（Node.js）
リポジトリに **scripts/check-links.js** を用意しています。  
**ski-powder-hunter.html** から RESORTS を読み取り、各 `url` に HEAD リクエストを送って応答ステータスを確認します。

```bash
cd POWDER
node scripts/check-links.js
```

- **200** など成功系 → リンク有効の目安。
- **404 / 403 / タイムアウト** → 要確認・修正候補として表示されます。
- `url` が空のゲレンデはスキップされます。

※ サーバー側の仕様で HEAD を拒否するサイトは「要確認」になることがあります。その場合はブラウザで開いて手動確認してください。

## ゲレンデ名の置き換え（名前変更）

ゲレンデ名を変更したときは、**次の3ファイル**をすべて同じ名前にそろえます。

| ファイル | 内容 |
|----------|------|
| RESORTS一覧.txt | 一覧テキスト |
| ski-powder-hunter.html | 日本語版の RESORTS データ |
| ski-powder-hunter-en.html | 英語版の RESORTS データ |

### 推奨: スクリプトで一括置換

**scripts/update-resorts-list.ps1** を実行すると、上記3ファイルを UTF-8 で読み書きして一括置換できます。今後も名前の変更はこのスクリプトで同じように対応できます。

```powershell
# リポジトリのルート（POWDER フォルダ）で実行
.\scripts\update-resorts-list.ps1 -Old "旧ゲレンデ名" -New "新ゲレンデ名"
```

例:
```powershell
.\scripts\update-resorts-list.ps1 -Old "サンアルピナ鹿島槍スキー場" -New "鹿島槍スキー場 ファミリーパーク"
```

- 旧名称が存在するファイルだけが更新され、見つからないファイルはスキップされます。
- RESORTS一覧.txt は UTF-8（BOM 付き）、HTML は UTF-8（BOM なし）で保存されます。

### RESORTS一覧.txt を手動で編集する場合（参考）

**RESORTS一覧.txt** は **UTF-8（BOM 付き）** です。エディタの既定が Shift_JIS などだと置換に失敗することがあります。

**原因**: ファイルは UTF-8 なのに別エンコードで開いていると、同じ見た目でもバイト列が一致せず「文字列が見つからない」になります。

**対処**: エディタで「UTF-8 で開く／保存」を指定するか、上記スクリプトを使ってください。

## リンク修正の反映
1. **ski-powder-hunter.html** を開く。
2. **RESORTS** の該当ゲレンデの **`"url": "現在のURL"`** を **`"url": "正しいURL"`** に書き換えて保存。
3. GitHub にプッシュするか、手動アップロードで反映。
